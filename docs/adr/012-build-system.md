# 012 — Build System

**Date**: 2026-05-17
**Status**: Accepted
**Origin**: AI recommended CMake as the only realistic choice given Drogon's ecosystem. Developer chose Ninja over Make as the build backend after understanding that Ninja's automatic parallelism is faster on a 2-core VPS and requires no flags to enable.

## Context

Three C++ services (core-api, uploader-api, watcher-api) each need a reproducible build system. The choice affects developer experience, CI/CD pipeline commands, Docker build time, and how Drogon's scaffolding tooling integrates.

## Options Considered

**CMake + Make**
- CMake is the standard for Drogon — `drogon_ctl` generates `CMakeLists.txt` automatically
- Make executes builds; parallelism requires explicit `-jN` flag every time
- Pre-installed on all Linux systems
- Slower than Ninja on multi-core machines

**CMake + Ninja**
- Same CMake config, different backend (`-G Ninja`)
- Ninja is designed for speed: always parallel, smarter incremental dependency tracking
- Noticeably faster cold and incremental builds on both VPS (2 vCPU) and dev Mac (8–12 cores)
- One extra install: `apt install ninja-build`
- `build.ninja` files are generated by CMake — not meant to be read or edited by hand

**Meson**
- No native Drogon support — build config must be written from scratch
- Not worth the friction when CMake + Ninja covers the need

**Bazel**
- Overkill; steep learning curve with no benefit for a 3-service project

**Raw Makefile**
- Educational but impractical — every compiler flag and dependency link written by hand
- CMake handles this correctly with less effort

## Decision

**CMake + Ninja**

CMake is effectively required by Drogon. Ninja is the better backend: automatic parallelism on all cores without any flags, faster builds, and cleaner error output. On a 2-core VPS, Ninja roughly halves build time compared to single-threaded Make.

## Build Commands (per service)

```bash
# Configure (once, or after CMakeLists.txt changes)
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug    # development
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Release  # production

# Build
ninja -C build

# Clean
ninja -C build -t clean
```

## Consequences

- `ninja-build` must be installed on dev machine and in CI runner (`apt install ninja-build`)
- Docker build images for C++ services must include both `cmake` and `ninja-build`
- `build/` directory is gitignored in all three service directories
- Both Debug and Release CMake configs must be defined — Debug for local dev, Release for Docker prod images
- Drogon's `drogon_ctl create project` generates the initial `CMakeLists.txt` — use it as the starting point
